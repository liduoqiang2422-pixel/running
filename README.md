# Demo Research Project

This demo project studies image restoration and digital watermark robustness.

- Training entrypoint: train.py
- Evaluation entrypoint: test.py
- Dataset root: data/demo_train
- Validation dataset: data/demo_val
- Checkpoint path: checkpoints/stage1/model_best.pth

The repository uses YAML configs under options/ and reports PSNR, SSIM, bit accuracy, and tamper localization.
这个项目可以定义为一个面向图像恢复与数字水印研究场景的本地优先、多 Agent 实验编排系统。它的目标不是单纯“帮用户读代码”或“给出训练命令建议”，而是把研究代码仓库从“信息分散、很难复现、出了问题只能人工排查”的状态，转化成一个可扫描、可规划、可验证、可诊断、可交付的自动化闭环。用户只需要给出项目路径和实验目标，系统就会自动读取仓库中的 README、依赖文件、训练脚本、测试脚本和 YAML 配置，识别训练入口、评测入口、数据集声明、checkpoint 关系、可能的评估指标以及项目标签，然后基于这些结构化事实生成完整的实验执行计划，并输出适合 GitHub 交付的 readiness report 和 benchmark handoff 信息。

这个项目的核心价值，在于它解决的不是某一个点状问题，而是研究实验工程化中的整条链路问题。很多图像恢复和水印项目都存在相同痛点：仓库结构不统一、环境依赖写法混杂、训练与评测入口分散、数据集路径往往藏在 README 或配置角落、checkpoint 继承关系不明确、实验失败后只能靠人一行一行翻日志、最终结果也缺少统一格式，导致论文复现成本很高、多人协作时交接成本更高。这个项目把这些隐性成本前移成显性的、可机器消费的工作流，把“看懂仓库”“判断能不能跑”“知道先修什么”“给出最终交付材料”统一纳入一个编排框架中。

它是一个有明确角色分工、共享上下文、状态流转和审批门控的多 Agent 系统。项目入口和使用说明在 README.md，命令行入口在 cli.py，调度核心在 orchestrator.py，角色定义集中在 agents.py，项目扫描逻辑在 project_inspector.py，状态和数据模型在 models.py。这意味着它不是靠一段大而全的流程函数硬编码实现，而是把扫描、规划、环境判断、执行规划、失败诊断、结果发布、人工审批拆成了独立的职责单元，具备清晰的扩展边界。
项目解决的核心痛点

解决研究仓库“能看不能跑”的问题。很多论文仓库表面上有 train.py 和 test.py，但真正决定能否复现的是依赖文件、配置文件、数据集声明和 checkpoint 传递关系。这个项目会优先把这些关键信息结构化提取出来，而不是停留在自然语言描述层面。

解决实验入口分散、缺少统一执行路径的问题。不同仓库可能同时有 Python、Shell、PowerShell 入口，也可能把关键参数放在 YAML 配置里。这个项目会自动推断环境初始化命令、训练命令和测试命令，并把它们统一到一个可审计的工作流里。

解决错误排查严重依赖资深成员经验的问题。传统流程里，实验失败通常意味着去看日志、猜环境、猜路径、猜权重、猜配置。这个项目引入诊断 Agent，将缺失训练入口、缺失评测入口、数据集声明不清、环境引导缺失、checkpoint lineage 缺失等问题转化为显式 issue 和 recommended fixes，降低排障对个人经验的依赖。

解决结果难交付、难复用的问题。研究型项目经常缺少一个标准化的“当前仓库准备好没有”的结论输出。这个项目会在最后生成包含项目资产、执行命令、指标目标、问题列表、修复建议和 readiness score 的报告，便于团队内部交接和对外 GitHub 展示。

解决自动化流程缺少人类把关的问题。很多自动化系统一旦跑到底，就默认完成，但研究实验和工程接入常常需要最后的人类确认。这个项目专门设计了 approval gate，在真正标记完成前保留人工审批节点，更符合实际研发流程。

核心逻辑流程

Scanner Agent 先扫描目标项目，读取 README、依赖文件、训练脚本、测试脚本和配置文件，提取项目快照，包括训练入口、评测入口、配置文件、数据集提示、checkpoint 引用和任务标签。

Planner Agent 基于扫描结果，把模糊的“实验目标”展开成一个显式工作流，不是单轮输出建议，而是生成一个带依赖关系的任务图。当前实现把流程拆成环境检查、执行规划、执行验证、失败诊断、结果发布和最终审批几个阶段。

Environment Agent 与 Execution Agent 可以并行工作。前者判断仓库如何初始化环境，例如 requirements、env.yaml 或 pyproject 的引导方式；后者一边准备训练与测试命令，一边定义校验步骤和预期产物。并行化的意义在于，环境判断和执行规划不必互相阻塞，提升整个流程吞吐。

Execution Agent 在验证阶段不会直接假装“已经训练完成”，而是进行可执行性模拟与 readiness 检查。它会判断训练入口是否存在、评测入口是否存在、数据集路径是否有声明、环境引导命令是否齐备，并据此给出 ready 或 needs_attention 状态。

Diagnoser Agent 会基于前面几个 Agent 产出的结构化结果做二次推理，把问题从“状态异常”翻译成“具体可修复事项”。例如训练入口缺失就建议暴露 train.py 或 train.sh，评测入口缺失就建议补 test.py 或 eval.py，数据集路径不明确就建议在 README 或 YAML 中声明，checkpoint 关系未记录就建议补充 resume 或 pretrained weight 说明。

Reporter Agent 汇总扫描资产、环境命令、训练命令、测试命令、目标指标、问题与修复建议，生成统一格式的 benchmark readiness report，并给出适合GitHub 交接的仓库材料列表。

Approval Agent 在最后一步执行人工门控，只有在 readiness score 达到阈值且审批通过后，整个工作流才会标记完成。这一设计让系统同时兼顾自动化与可控性。

是否包含长推理
包含，但更准确地说，这是“显式的长链任务推理”而不是黑盒式的一次性回答。它的长推理体现在三个层面。

第一，它会先把输入目标转成跨阶段任务图，而不是直接输出一句建议。也就是说，系统不是“看到项目路径就给一个命令”，而是先扫描事实，再基于事实生成后续阶段，这是一种多步展开式推理。

第二，每个 Agent 都会消费上游 Agent 的结构化结果，而不是在真空中各自工作。扫描结果进入共享记忆后，规划 Agent 利用它生成 DAG，环境 Agent 和执行 Agent 进一步基于这些信息产出自己的判断，诊断 Agent 再把多个中间结果合并成结论，报告 Agent 最后做总汇总。这种逐层依赖的过程，本质上就是一个跨阶段的 reasoning chain。

第三，整个推理链是可审计、可恢复、可中断续跑的。项目把任务状态、产物、记忆和历史事件保存在统一状态模型里，不是临时对话上下文，因此它更适合工程化落地。对应状态流转与不可变数据模型可以在 models.py 和 memory.py 中看到。

多 Agent 如何高效协作

角色清晰。每个 Agent 只处理自己负责的一小段职责，避免单个“大脑”既扫描项目、又推断命令、又分析问题、又汇总报告，降低系统复杂度。共享记忆。所有关键中间结果都会写入 shared memory，下游 Agent 直接读取最新版本的数据，不需要重复扫描和重复解析，减少冗余计算。

DAG 调度。任务不是死板串行，而是按依赖关系调度。只要依赖满足，任务就能进入 ready 阶段，从而支持真正意义上的并发执行，而不是表面上的“多 Agent”名义串行。

并发控制。调度器通过 max_concurrency 控制同一波次的执行数量，既保证效率，也给后续接入真实工具执行器时保留资源约束能力。

人机协同。审批门控意味着系统不会在最后一步绕过人类确认，这一点对科研实验、生产环境接入、GitHub 公开交付都很重要。

可测试性。这个项目不是只写了主逻辑，没有验证。当前已经有完整工作流、审批等待、审批后恢复三个单测场景，见 test_orchestrator.py。这说明它不仅能描述流程，也能验证流程真的按设计工作。
